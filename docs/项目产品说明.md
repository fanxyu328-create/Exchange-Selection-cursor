# ExchangeSelect · 交换生选校系统

> 一个支持多轮、多设备协同的选校系统，从 0 到上线全程用 **Vibe Coding** 与 **产品思维** 推进，适合作为 AI 产品 / 全栈向的作品展示。

---

## 一、产品在解决什么问题？

**场景**：学校组织交换生选校——按排名依次选校、每人可选多轮（如秋季 + 春季），名额有限、需实时看到剩余名额和当前轮到谁。

**痛点**：
- 多人、多设备要看到同一份「当前轮到谁」「各校剩余名额」「每人选了什么」；
- 管理员要能初始化/重置名单与学校表，且操作要同步到所有人；
- 若只做本地 Demo，无法体现「真实可用的协作产品」。

**目标**：做一个**真正可多人多设备使用、数据实时同步**的选校产品，而不是单机 Demo。

---

## 二、产品与功能概览

- **角色**：学生（按排名选校/跳过）、管理员（初始化数据、查看状态）。
- **核心流程**：登录 → 看当前轮次与剩余名额 → 轮到自己时选校或跳过 → 多轮（如 Round 1 秋季、Round 2 春季）自动推进。
- **管理端**：
  - 进入页面**默认拉取后端当前数据**展示在表单中（非写死的 Demo）；
  - 支持 **下载 CSV 模版**、**上传 CSV 解析** 填入用户/学校数据；
  - **Reset & Initialize**：先清空再写入，保证前后端完全一致。
- **数据与同步**：
  - 有 Supabase 配置时：**云端数据库 + 轮询 + Realtime 订阅**，名额与状态多端自动同步；
  - 无配置时：本地 localStorage，便于本地演示与开发。

技术栈：React + TypeScript + Vite，后端/同步层 Supabase（PostgreSQL + Realtime），部署在 Vercel。

---

## 三、产品思维体现在哪？

1. **从「能跑」到「能用」**  
   先做出本地版验证流程，再接入后端与同步，保证多人、多设备真实可用，而不是停留在单机 Demo。

2. **管理端体验**  
   - 管理员进入即看到**当前库里的数据**，而不是固定示例，减少误操作、符合「编辑现有数据」的心智。  
   - CSV 模版 + 上传解析，降低批量维护门槛，适合运营/教务使用。

3. **数据一致性**  
   管理员重置时采用「先清空再写入」，避免旧数据残留导致前后端不一致，把一致性当作产品约束而不是实现细节。

4. **渐进式复杂度**  
   支持「无后端本地跑」与「配置 Supabase 后多端同步」两种模式，便于演示、评审和迭代。

## 四、Vibe Coding 如何参与？

本项目从需求到上线，全程在 **Cursor** 中通过自然语言需求 + 小步迭代完成，体现「和 AI 一起做产品」的节奏：

- **需求级描述**：如「支持管理员下载 CSV 模版、上传 CSV 解析」「管理员进入页面默认显示后端当前数据」「前端的改动要同步到后端、名额和状态要自动同步」——由 AI 完成方案设计、接口设计与实现。
- **架构级决策**：如「要多人多设备同步是否需要后端、如何实现」——在对话中确定用 Supabase，并完成建表、RLS、前端统一数据层与 Realtime/轮询策略。
- **细节与修 bug**：如「管理员重置后后端没有完全同步」「学校剩余名额和同学状态不会自动同步」——通过描述现象，由 AI 定位到「先清空再写入」和「Supabase 下轮询 + Realtime 双保险」并改代码。
- **上线与配置**：从「部署到 Vercel」「在 Supabase 里要做什么」到「.env 不提交、在 Vercel 上如何配环境变量」，均用说明 + 文档（如 `docs/SUPABASE_SETUP.md`）沉淀，便于复现和展示。

整体节奏是：**产品目标 → 功能列表 → 分步实现 → 上线与运维说明**，代码与文档同步推进，适合作为「Vibe Coding + 产品思维」的样本。
